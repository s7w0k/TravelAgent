# 动态摘要与上下文压缩引擎实施方案

## 1. 目标说明

本方案对应 `需求文档 5：动态摘要与上下文压缩引擎`，基于当前项目已经具备的：

- WebSocket 对话主链路
- 会话持久化能力
- 长期记忆存储能力
- RAG 检索上下文注入能力

构建一套**最小改动、可运行、与现有接口兼容**的上下文压缩方案。

当前现状：

- `conversation_history` 仅按条数截断
- 历史消息超过窗口后会直接丢弃
- 尚无历史摘要池
- 尚无结构化关键约束保留层
- 输入模型时尚未形成分层上下文拼装

因此，本次实现目标是：

1. 建立滑动窗口机制
2. 对窗口外历史消息生成规则式动态摘要
3. 结构化保留不可丢失的关键约束
4. 在调用模型时分层拼装上下文
5. 保证整体功能可行且正常运行

---

## 2. 当前代码结构与切入点

### 2.1 `backend/main.py`
当前负责：

- WebSocket 收发
- 加载历史会话
- 构建输入消息
- 管理 `conversation_history`

这里是上下文压缩的主接入点。

### 2.2 `backend/session_memory.py`
当前已负责：

- 会话持久化
- 长期记忆抽取与读取
- 用户结构化偏好存储

这里最适合继续承接：

- 历史摘要保存
- 结构化关键约束抽取
- 压缩后的会话视图构建

### 2.3 当前缺失能力
缺少：

- 滑动窗口之外的历史摘要
- 会话级关键约束保留
- 上下文压缩后的消息拼装

因此，本次主要在 `session_memory.py` 和 `main.py` 内增强。

---

## 3. 实现原则

### 原则 1：不改 Agent 主逻辑
不改：

- Multi-Agent 图
- Planner / Writer / Search / Visualization Agent 内部逻辑
- RAG 检索逻辑

### 原则 2：优先规则式摘要，不引入额外模型依赖
当前版本不引入专门摘要模型，避免：

- 依赖膨胀
- 成本上升
- 推理链路更复杂

因此本次采用：

- 规则式摘要
- 结构化关键信息抽取
- 分层上下文拼装

### 原则 3：摘要与结构化约束分开保存
因为“可读摘要”和“不可丢失约束”是两类信息：

- 摘要：用于压缩历史语义
- 结构化约束：用于确保目的地/预算等不丢

### 原则 4：保持旧客户端兼容
即使客户端不感知摘要机制，系统仍可正常工作。

---

## 4. 总体架构

目标结构：

```text
历史会话消息
   -> 滑动窗口切分
   -> 窗口外消息生成摘要
   -> 提取关键约束
   -> 保存到 session
   -> 新一轮调用时分层拼装：
      当前问题
      + 最近窗口消息
      + 长期记忆
      + 会话关键约束
      + 历史摘要
      + RAG 上下文
```

---

## 5. 数据模型设计

### 5.1 SessionRecord 扩展字段
在现有 session JSON 中增加：

- `summary`
- `structured_context`

其中：

#### `summary`
保存窗口外历史消息的压缩文本

#### `structured_context`
保存不可丢失信息，例如：

- destination
- departure_city
- budget_max
- days
- persona
- accommodation_preferences
- food_avoidances
- confirmed_items

---

## 6. 滑动窗口设计

### 6.1 目标
避免完整历史消息不断累积进入模型上下文，导致：

- token 消耗过高
- 旧消息噪音过多
- 关键约束容易被淹没

### 6.2 方案
定义两个层级：

1. **原始会话完整保存**：session 文件仍保存完整消息
2. **模型输入窗口压缩**：给模型时只传最近窗口消息

### 6.3 建议窗口
新增配置：

- `CONTEXT_WINDOW_MESSAGES`
- `ENABLE_DYNAMIC_SUMMARY`
- `ENABLE_CONTEXT_COMPRESSION`

其中：

- 最近 `N` 条消息保留原文
- 更早消息进入摘要层

---

## 7. 动态摘要设计

### 7.1 当前版本目标
对窗口外历史消息生成可读且可复用的摘要文本。

### 7.2 实现方式
规则式摘要，按用户消息和助手消息分别提取：

- 用户提出了哪些核心约束
- 系统已经给出了哪些关键方案
- 已确认过哪些偏好

### 7.3 摘要格式
类似：

```text
## 历史对话摘要
- 用户曾询问：上海到苏州三日游、预算 3000、亲子出行
- 已提及偏好：住地铁附近、不吃辣
- 系统曾输出：苏州经典三日游方案与交通建议
```

### 7.4 更新时机
每轮消息完成后：

1. 以完整历史为输入
2. 计算窗口外消息
3. 生成最新摘要
4. 回写到 session 文件

---

## 8. 结构化关键约束保留设计

### 8.1 目标
确保即使原始历史被压缩，以下信息仍不丢：

- 目的地
- 日期/时长
- 预算
- 人群画像
- 饮食禁忌
- 住宿偏好
- 已确认交通/住宿/景点

### 8.2 实现方式
在 `session_memory.py` 中新增会话级抽取函数：

- 从完整历史消息中提取核心约束
- 优先复用已有长期记忆规则
- 再补充天数、确认项等会话级字段

### 8.3 存储位置
写入 session JSON 的 `structured_context` 字段。

---

## 9. 分层上下文拼装设计

### 9.1 当前问题
现有模型输入是：

- system prompt
- 全部 `conversation_history`
- 当前用户消息
- RAG 上下文

问题在于：

- 长对话会越来越长
- 没有层次化组织

### 9.2 目标拼装顺序
每次调用模型时按优先级注入：

1. 系统提示词
2. 当前会话关键约束
3. 用户长期记忆
4. 历史摘要
5. RAG 上下文
6. 最近窗口消息
7. 当前用户消息

### 9.3 预期效果
这样即使窗口较短，也能保留：

- 历史主要语义
- 关键硬约束
- 个性化偏好
- 最新对话上下文

---

## 10. 与现有代码的具体改动点

### 改动点 1：`backend/config.py`
新增少量配置：

- `ENABLE_DYNAMIC_SUMMARY`
- `ENABLE_CONTEXT_COMPRESSION`
- `CONTEXT_WINDOW_MESSAGES`

### 改动点 2：`backend/session_memory.py`
新增：

- 会话摘要生成
- 会话结构化上下文抽取
- 压缩上下文视图构建
- 保存 session 时回写 `summary` 与 `structured_context`

### 改动点 3：`backend/main.py`
新增：

- 加载压缩上下文视图
- 构建分层上下文消息
- 只把最近窗口消息直接传给模型

### 不改动模块
本次不改：

- `backend/rag/*`
- `backend/agents/*`
- 前端组件
- API 主业务逻辑

---

## 11. 兼容性说明

本方案保证：

- 现有 WebSocket 输入格式不变
- 会话持久化接口不变
- 旧 session 文件缺少 `summary/structured_context` 也能兼容读取
- 未启用压缩时仍可退回原逻辑

---

## 12. 风险与规避

### 风险 1：规则式摘要不够自然
规避：
- 先保证可读、稳定、低风险
- 后续再升级 LLM 摘要

### 风险 2：重要信息被摘要遗漏
规避：
- 单独保留结构化关键约束
- 摘要只做语义压缩，不承担全部记忆职责

### 风险 3：上下文顺序变化影响结果
规避：
- 保留最近窗口消息原文
- 仅把旧消息压缩到摘要层

### 风险 4：旧会话文件结构不一致
规避：
- 读取时提供默认值
- 兼容缺字段情况

---

## 13. 本次实际实施范围

本次将实现：

- 滑动窗口
- 规则式动态摘要
- 会话级结构化关键约束提取
- 分层上下文拼装
- session 文件持久化 `summary` 与 `structured_context`

本次不实现：

- 基于 LLM 的高质量摘要
- token 级精确预算控制
- 多阶段摘要池分层管理
- 历史规划自动结构化成完整草案对象

---

## 14. 结论

基于当前项目代码结构，最合理的落地方式不是引入新的上下文管理框架，而是：

1. 以 `session_memory.py` 为摘要与压缩中心
2. 以 `main.py` 为分层上下文拼装入口
3. 用规则式摘要 + 结构化关键约束，先完成第一版可用能力
4. 保持现有系统稳定、接口兼容、运行可控

该方案改动集中、依赖最少、易于验证，适合作为“动态摘要与上下文压缩引擎”的当前版本实现方案。
