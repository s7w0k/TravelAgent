# 会话持久化与长期记忆中心实施方案

## 1. 目标说明

本方案对应 `需求文档 4：会话持久化与长期记忆中心`，基于当前项目已有的 WebSocket 对话与 Multi-Agent 执行链路，给出一套**最小改动、可运行、与现有接口兼容**的落地方案。

当前现状：

- 对话历史仅保存在 `websocket_endpoint()` 的内存变量 `conversation_history`
- WebSocket 断开后历史即丢失
- 尚无 `session_id / user_id` 维度的持久化存储
- 尚无关键用户偏好与约束的结构化长期记忆

因此，本次实现目标是：

1. 在不改动核心 Agent 编排逻辑的前提下
2. 增加本地文件级会话持久化能力
3. 增加规则式关键记忆抽取与读取能力
4. 支持按 `session_id / user_id` 查询和恢复历史会话
5. 保证整体功能可行且正常运行

---

## 2. 当前代码结构与切入点

### 2.1 `backend/main.py`
当前负责：

- WebSocket 建连
- 接收用户输入
- 维护内存级 `conversation_history`
- 调用 Agent 执行与流式返回

该文件是本次会话持久化的主接入点。

### 2.2 `backend/schemas.py`
当前已有：

- `ConversationMessage`

可继续作为消息结构基础，不需要改动。

### 2.3 当前缺失能力
缺少：

- 会话存储器
- 长期记忆存储器
- 记忆提取器
- 对外查询接口

因此本次新增模块应集中承接这些职责。

---

## 3. 实现原则

### 原则 1：不改动不相关业务模块
本次不改：

- Agent 内部实现
- RAG 检索链路
- 前端主要交互逻辑
- Multi-Agent 状态图

### 原则 2：先做本地轻量持久化
当前阶段不引入数据库，优先使用本地 JSON 存储：

- 部署简单
- 无新增重量级依赖
- 便于快速验证

### 原则 3：规则式记忆抽取优先
当前版本不引入 LLM memory extractor，而采用规则抽取：

- 出发地
- 目的地
- 预算范围
- 住宿偏好
- 饮食禁忌
- 出行人群类型

### 原则 4：兼容现有 WebSocket 消息格式
如果客户端未显式传 `session_id / user_id`：

- 系统自动生成 `session_id`
- `user_id` 可为空
- 不影响现有流程运行

---

## 4. 总体架构

目标结构：

```text
WebSocket 消息
   -> 解析 session_id / user_id
   -> 加载历史会话
   -> 加载用户长期记忆
   -> 拼装输入上下文
   -> Agent 执行
   -> 保存消息
   -> 抽取关键记忆
   -> 更新用户记忆
```

并新增只读 REST 接口：

- 按 `session_id` 获取会话详情
- 按 `user_id` 获取会话列表
- 按 `user_id` 获取长期记忆

## 5. 数据模型设计

### 5.1 SessionRecord
字段：

- `session_id`
- `user_id`
- `created_at`
- `updated_at`
- `messages`
- `summary`（当前版本可为空）
- `last_plan`（当前版本可为空）

### 5.2 UserMemory
字段：

- `user_id`
- `departure_city`
- `destinations`
- `budget_min`
- `budget_max`
- `accommodation_preferences`
- `food_preferences`
- `food_avoidances`
- `travel_personas`
- `updated_at`

### 5.3 存储方式
使用本地目录：

- `data/session_memory/sessions/{session_id}.json`
- `data/session_memory/users/{user_id}.json`

并增加 `user_sessions.json` 作为用户到会话列表的映射索引。

## 7. 会话恢复设计

### 7.1 WebSocket 输入兼容格式
当前消息支持 JSON，因此扩展解析：

```json
{
  "message": "帮我规划上海到苏州三日游",
  "session_id": "sess_xxx",
  "user_id": "user_001"
}
```

### 7.2 恢复逻辑
当建立或继续会话时：

1. 若传入 `session_id` 且存在历史记录，则加载历史消息
2. 若传入 `user_id`，则加载用户长期记忆
3. 将长期记忆转成系统上下文片段注入输入消息
4. 再继续后续 Agent 执行

### 7.3 无 session_id 时
- 自动生成新的 `session_id`
- 首次向客户端返回该值，便于前端后续复用

---

## 8. 对外查询接口设计

新增 REST 接口：

### 8.1 获取会话详情
`GET /memory/sessions/{session_id}`

返回：

- session 基本信息
- 全部消息记录

### 8.2 获取用户会话列表
`GET /memory/users/{user_id}/sessions`

返回：

- 该用户的会话 ID 列表
- 最近更新时间

### 8.3 获取用户长期记忆
`GET /memory/users/{user_id}/memory`

返回：

- 当前用户结构化偏好与长期记忆

---

## 9. 与现有代码的具体改动点

### 改动点 1：新增 `backend/session_memory.py`
负责：

- 会话存储
- 用户记忆存储
- 记忆提取
- 会话恢复

### 改动点 2：新增 `backend/memory_api_routes.py`
负责：

- 会话查询接口
- 用户记忆查询接口

### 改动点 3：修改 `backend/main.py`
只在 WebSocket 流程中增加：

- 解析 `session_id / user_id`
- 加载历史会话
- 注入长期记忆
- 每轮消息后持久化
- 返回 `session_meta`

### 改动点 4：修改 `backend/config.py`
新增少量配置：

- 会话存储目录
- 是否启用持久化
- 是否启用长期记忆

## 10. 兼容性说明

本方案保证：

- 现有 WebSocket 纯文本消息仍可继续使用
- 未传 `session_id / user_id` 时仍可正常对话
- 现有 Agent / RAG / API 主流程不受影响

也就是说，本次属于：

- **内部增强**
- **对现有调用基本兼容**

---

## 11. 风险与规避

### 风险 1：本地文件写入失败
规避：
- 初始化时自动创建目录
- 单次保存失败不影响主对话流程
- 错误仅记录日志

### 风险 2：历史消息过长
规避：
- 当前先保留完整 session 消息
- 输入模型时仍只使用现有窗口长度限制
- 后续可再接“动态摘要模块”

### 风险 3：规则抽取不完整
规避：
- 允许记忆为空
- 只增不破坏主流程
- 后续可升级为 LLM 抽取

### 风险 4：会话恢复影响旧客户端
规避：
- 输入字段全部可选
- 系统自动生成默认 session_id

---

## 12. 本次实际实施范围

本次将实现：

- 会话 JSON 持久化
- 用户长期记忆 JSON 存储
- 规则式关键偏好抽取
- 会话恢复
- 会话 / 记忆查询接口

本次不实现：

- 数据库存储
- 多设备同步冲突解决
- 自动摘要压缩
- 计划草案结构化版本管理
- 基于 LLM 的高精度记忆抽取

---

## 13. 结论

基于当前项目结构，最合理的实现方式不是直接引入数据库或复杂记忆框架，而是：

1. 以 `main.py` 的 WebSocket 链路为会话接入点
2. 新增独立的 `session_memory` 模块承接持久化与长期记忆职责
3. 用本地 JSON 先完成最小可用能力
4. 保持现有系统整体稳定运行

该方案改动范围集中、风险可控、可直接支撑“会话持久化与长期记忆中心”的第一版落地。
