1. 项目已经实现了一个基础版的多 Agent 旅行规划系统；

# 需求文档 1：混合检索引擎升级

## 1. 需求名称
混合检索引擎升级（Milvus + BM25 + Hybrid RAG）

## 2. 背景
当前项目仅实现了基于 Chroma 的向量检索，无法充分覆盖旅行规划场景中大量“关键词强相关、结构化条件明确、长尾表达丰富”的查询需求。对于“苏州园林三日游预算 2000”“上海迪士尼亲子攻略”“北京到苏州高铁二等座”等典型请求，仅依赖向量检索会造成召回不稳定、关键词遗漏、结果不够可控的问题。

## 3. 目标
建设面向旅行场景的混合检索能力，实现：
- 由 Chroma 过渡到 Milvus
- 引入 BM25 关键词召回
- 构建向量 + 关键词混合召回
- 支持结构化过滤，提高召回相关性与稳定性

## 4. 用户价值
- 提高目的地、交通、预算、玩法类问题的召回准确率
- 降低大模型“编造攻略”的风险
- 为后续重排、过滤、个性化推荐打基础

## 5. 适用场景
- 搜索城市攻略
- 搜索景点玩法
- 搜索交通信息
- 搜索预算建议
- 搜索特定用户人群玩法，如亲子、情侣、学生党

## 6. 核心功能
### 6.1 向量检索切换至 Milvus
- 支持文档 embedding 存储
- 支持高并发检索
- 支持 metadata 字段检索
- 支持后续扩展分区与多集合管理

### 6.2 BM25 检索
- 对文档标题、正文、标签建立关键词索引
- 支持 destination、spot、budget、season 等关键词强匹配

### 6.3 Hybrid RAG
- 同一 query 同时走向量召回和 BM25 召回
- 对双路结果进行融合去重
- 输出统一候选集供下游 Rerank

### 6.4 元数据过滤
支持按以下条件过滤：
- 目的地城市
- 旅行天数
- 预算区间
- 玩法标签
- 用户类型
- 季节/月份

## 7. 业务规则
- 默认优先召回与目的地强相关文档
- 当 query 中存在明确实体词时，BM25 权重高于纯语义召回
- 若用户条件明确，先过滤再召回
- 若用户条件不明确，先广召回后重排

## 8. 输入输出
### 输入
- 用户 query
- 结构化约束：目的地、预算、天数、偏好

### 输出
- 标准化候选文档列表
- 每条文档包含来源、标签、召回分数、召回类型

## 9. 验收标准
- 对 Top 20 常见旅行问题，召回相关文档覆盖率显著高于当前版本
- 对明确关键词类 query，BM25 召回结果命中率达到预期
- 支持 destination / budget / days 条件过滤生效

## 10. 优先级
P0

# 需求文档 2：召回优化链路

## 1. 需求名称
Query Rewrite / 去重 / Rerank / 条件过滤召回链路

## 2. 背景
当前系统检索链路较短，用户 query 直接进入向量检索，缺少查询改写、候选去重、结果重排与意图约束等关键步骤，导致：
- 模糊查询效果差
- 召回结果重复
- 候选相关性不稳定
- 对预算/天数等条件理解不充分

## 3. 目标
建立标准化检索增强流水线：
1. Query 解析
2. Query Rewrite
3. 多路召回
4. 去重
5. Rerank
6. 条件过滤
7. 输出高置信上下文

## 4. 核心功能
### 4.1 Query Rewrite
- 补全目的地别名，如“魔都=上海”
- 扩展玩法表达，如“遛娃=亲子”
- 将口语化需求改写为检索友好表达
- 生成多条候选子查询

### 4.2 去重
- 标题近似去重
- URL 或 source_id 去重
- 内容相似度去重

### 4.3 Rerank
- 结合 query、destination、days、budget、persona 对候选结果重排
- 优先保留攻略完整、时效性强、来源可信的结果

### 4.4 条件过滤
- 预算高于用户上限的候选降权
- 超出目标天数适配范围的候选降权
- 与目标城市不一致的候选剔除

## 5. 用户价值
- 搜得更准
- 搜得更全
- 减少重复垃圾结果
- 降低 Planner 生成失真行程的概率

## 6. 验收标准
- 同一 query 重复结果占比显著下降
- 模糊 query 的首屏结果相关性提升
- 有明确约束的 query，返回结果与约束一致率明显提高

## 7. 优先级
P0

- 超预算时能自动给替代方案
- 支持至少三档预算方案对比

## 7. 优先级
P1

# 需求文档 8：实时路线优化与行程可行性校验系统

## 1. 需求名称
实时路线优化与行程可行性校验系统

## 2. 背景
当前路线优化近似为占位实现，缺少真实地图耗时、营业时间、天气、交通联动约束。

## 3. 目标
构建“能执行”的旅行行程，而不是“看起来像计划”的文本。

## 4. 核心功能
### 4.1 路线优化
- 基于地图服务获取景点间耗时
- 优化游玩顺序
- 避免折返

### 4.2 开放时间校验
- 检查景点是否营业
- 检查预约要求
- 检查到达时间是否合理

### 4.3 天气联动
- 雨天/高温/极端天气触发调整建议
- 室内外景点自动替换

### 4.4 交通联动
- 火车/地铁/打车耗时纳入当日路线
- 到站时间影响首日行程安排
- 返程时间影响末日行程安排

## 5. 用户价值
- 行程更真实
- 减少不可执行安排
- 更像专业旅行顾问

## 6. 验收标准
- 生成行程具备时间可执行性
- 景点开放时间冲突可被识别
- 恶劣天气下有替代方案

## 7. 优先级
P0

# 需求文档 9：个性化旅行画像与推荐系统

## 1. 需求名称
个性化旅行画像与推荐系统

## 2. 背景
当前系统主要是通用规划，缺乏用户分层和偏好建模，导致生成结果趋同。

## 3. 目标
建立用户画像体系，让旅行推荐真正个性化。

## 4. 画像维度
- 消费层级：省钱 / 均衡 / 品质
- 出行人群：独行 / 情侣 / 亲子 / 老人 / 朋友
- 节奏偏好：特种兵 / 休闲慢游
- 内容偏好：美食 / 拍照 / 自然 / 历史 / 娱乐
- 交通偏好：公共交通 / 打车 / 自驾
- 住宿偏好：经济 / 舒适 / 高端

## 5. 核心功能
### 5.1 画像推断
从历史对话与本次输入推断用户类型

### 5.2 推荐策略
针对不同画像选择不同玩法模板

### 5.3 偏好记忆
用户明确表达的偏好进入长期记忆

### 5.4 个性化输出
同一城市同一天数，对不同用户生成不同方案

## 6. 用户价值
- 更懂用户
- 推荐更贴合
- 提升满意度与复用率

## 7. 验收标准
- 不同画像用户的推荐结果明显区分
- 偏好可跨会话继承
- 画像对排序与规划产生实质影响

## 8. 优先级
P1

# 需求文档 10：行程编辑、重规划与可执行闭环系统

## 1. 需求名称
行程编辑、重规划与可执行闭环系统

## 2. 背景
当前系统更偏一次性生成，缺少后续编辑与闭环执行能力。真实用户往往会不断修改方案。

## 3. 目标
让用户能围绕已有计划持续调整、重排、确认并进入执行阶段。

## 4. 核心功能
### 4.1 行程结构化编辑
支持用户发起：
- 替换某个景点
- 删除某天安排
- 增加休息时间
- 调整预算
- 更改出发日期

### 4.2 局部重规划
只重算受影响部分，而不是整份计划全部重生成

### 4.3 版本管理
- 保存多个行程版本
- 支持版本比较
- 支持回退

### 4.4 执行闭环
- 导出攻略
- 导出路线
- 导出清单
- 预留预订与提醒能力接口

## 5. 用户价值
- 计划可反复打磨
- 减少重来成本
- 更接近真实出游决策过程

## 6. 验收标准
- 行程支持结构化修改
- 修改后可局部重规划
- 历史版本可追溯

## 7. 优先级
P1
